# Kural — Infrastructure Setup Guide

> Sales conversation analysis platform for the South Indian market.
> Two-tier cloud architecture: Hetzner CPU coordinator + RunPod Serverless GPU worker.

---

## Overview

```
┌──────────────────────────────────┐     ┌──────────────────────────────────┐
│   Hetzner Cloud — always-on      │     │   RunPod — on-demand GPU          │
│                                  │     │                                  │
│  FastAPI coordinator             │     │  Audio preprocessing             │
│  Job queue (PostgreSQL)          │◄───►│  Speaker diarization (PyAnnote)  │
│  LLM insights (DeepSeek/OpenAI)  │     │  Transcription (AudioX Whisper)  │
│  React frontend                  │     │  Results → S3                    │
│  Hetzner Object Storage (S3)     │     │  Webhook → coordinator           │
└──────────────────────────────────┘     └──────────────────────────────────┘
```

**Why two tiers?**
- GPU servers cost ~10× more than CPU. We only need GPU for diarization and transcription.
- The GPU is idle between jobs — RunPod Serverless is pay-per-second with no idle cost.
- The CPU server handles everything else and stays on 24/7 at minimal cost.

## Quick Start — Day-to-day Operations

### Enter the ops container

All infrastructure commands (Terraform, Ansible, hcloud, aws CLI) run inside
the ops container. Never run these directly on your laptop — this ensures
everyone uses identical tool versions.

```bash
cd /path/to/kural
docker compose -f infra/docker-compose.ops.yml run --rm ops
```

You'll get a shell prompt: `[kural-ops] /ops $`

### SSH to the server

```bash
# From your laptop
ssh -i ~/.ssh/kural_ed25519 root@46.225.211.24

# From inside the ops container (key auto-loaded by entrypoint)
ssh root@46.225.211.24
```

### Redeploy after code changes

The fastest path: **just push to `main` or `cloud-deploy`**. GitHub Actions
auto-deploys to Hetzner (frontend/API changes) and/or rebuilds the worker image
(GPU code changes). See [CI/CD](#cicd--github-actions) below.

For manual redeploys from the ops container:

```bash
# Fast code-only redeploy (sync code + rebuild API + restart services)
./scripts/deploy.sh sync

# Full Ansible run (includes system phases — use after server reprovisioning)
./scripts/deploy.sh ansible
```

### Test S3 connectivity

```bash
# From inside the ops container
bash /ops/scripts/test-s3.sh
```

### Run Terraform (infrastructure changes)

```bash
# From inside the ops container
cd /ops/terraform
terraform plan     # preview changes
terraform apply    # apply changes
```

### Check Hetzner resources

```bash
# From inside the ops container
hcloud server list
hcloud volume list
hcloud ssh-key list
hcloud firewall list
```

### Check RunPod endpoint health

```bash
# From inside the ops container
curl -s -H "Authorization: Bearer $RUNPOD_API_KEY" \
  "https://api.runpod.ai/v2/$RUNPOD_ENDPOINT_ID/health" | jq .
```

```bash
cp infra/.env.ops.example infra/.env.ops
# Edit infra/.env.ops — fill in all values (see .env.ops.example for field descriptions)

terraform apply
```

### 7. Configure the server

```bash
# Still inside the ops container
./scripts/deploy.sh ansible
```

## CI/CD — GitHub Actions

Two workflows handle automated deployment. Neither requires manual intervention
after a `git push`.

### Server Deploy (`deploy-server.yml`)

Deploys the **frontend + API** to the Hetzner server. Equivalent to
`deploy.sh sync` but fully automated.

**Triggers:**
- Push to `main`/`cloud-deploy` touching: `frontend/`, `src/kural/api/`,
  `src/kural/insights/`, `src/kural/models.py`, `src/kural/config.py`,
  `src/kural/db.py`, `docker/Dockerfile.api`, `docker/Dockerfile.web`,
  `docker/Caddyfile`, `docker/docker-compose.prod.yml`, `requirements/api.txt`,
  `requirements/base.txt`, `pyproject.toml`, `infra/deploy.conf`
- Manual dispatch via GitHub → Actions → "Deploy to Server" → "Run workflow"

**What it does:** rsync code → `docker compose build api web` →
`docker compose up -d` → health check.

**No GitHub Secrets needed.** The SSH key and server IP are committed to the
private repo:

| File | Purpose |
|---|---|
| `infra/deploy.conf` | `SERVER_IP=` — edit this when the server changes |
| `infra/deploy-keys/kural_ed25519` | SSH private key |
| `infra/deploy-keys/kural_ed25519.pub` | SSH public key |

### Worker Image Build (`build-worker.yml`)

Builds the **GPU worker image**, pushes to GHCR, and auto-updates the RunPod
template.

**Triggers (worker-specific paths only):**
- Push to `main`/`cloud-deploy` touching: `src/kural/worker/`,
  `src/kural/audio/`, `src/kural/config.py`, `src/kural/storage.py`,
  `src/kural/insights/`, `docker/Dockerfile.worker`, `requirements/worker.txt`,
  `requirements/base.txt`, `pyproject.toml`, `.dockerignore`
- Manual dispatch via GitHub → Actions → "Build Worker Image" → "Run workflow"

> **Note:** Server-only files like `src/kural/api/`, `src/kural/models.py`,
> `src/kural/db.py`, and `frontend/` do **not** trigger a worker rebuild.

**On success:** The workflow patches the RunPod template via REST API to use
the exact SHA-tagged image.

**GitHub Secrets required (worker builds only):**

| Secret | Description |
|---|---|
| `RUNPOD_API_KEY` | RunPod API key |
| `RUNPOD_TEMPLATE_ID` | RunPod template ID (`aqtmr0o476`) |
| `HF_TOKEN` | HuggingFace token (passed as BuildKit secret — never stored in image) |

## GPU Worker — RunPod Serverless

### Architecture

The GPU worker runs as a RunPod Serverless function. It is stateless per-invocation but maintains **warm state** (loaded models) between jobs on the same container instance.

```
Cold start:
  Container launch → Python imports → ModelRegistry.preload_all()
    → PyAnnote loads to GPU (~25s)
    → AudioX Whisper loads to GPU (~15s)
  → runpod.serverless.start() → waiting for jobs

Warm job:
  Job arrives → handler() → pipeline.process_job()
    [1/5] Download audio from S3
    [2/5] Preprocess (noise reduction, silence removal)
    [3/5] Diarize (PyAnnote — 0s load, already in GPU)
    [4/5] Transcribe segments (AudioX Whisper — 0s load)
    [5/5] Upload results.json to S3
  → webhook → coordinator → LLM insights → job done
```

### Model Baking

Models are downloaded at **image build time** using `huggingface_hub.snapshot_download()` and saved to the HuggingFace cache (`/root/.cache/huggingface`) inside the image. At runtime, models load from disk — no download needed.

**Baked models:**
- `pyannote/speaker-diarization-3.1` (speaker diarization pipeline)
  - Internal deps also baked: `pyannote/segmentation-3.0`, `pyannote/wespeaker-voxceleb-resnet34-LM`
- `jiviai/audioX-south-v1` (AudioX Whisper for South Indian languages)

### RunPod Registry Credential

The worker image is private (GHCR). RunPod needs a credential to pull it.

1. Create a GitHub PAT with `read:packages` scope at https://github.com/settings/tokens
2. In RunPod Console → Serverless → Registry Credentials → add credential with:
   - Registry: `ghcr.io`
   - Username: your GitHub username (lowercase)
   - Password: the PAT
3. In the RunPod template, set the container image to the GHCR image URL and select the credential
4. **When the PAT expires:** create a new PAT, update the RunPod registry credential entry — no rebuild needed

### Manual Template Update (if CI fails)

```bash
# From ops container — patch template to use a specific image tag
curl -s -X PATCH \
  -H "Authorization: Bearer $RUNPOD_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"imageName": "ghcr.io/spartan-velanjeri/kural-worker:sha-XXXXXXX"}' \
  "https://rest.runpod.io/v1/templates/$RUNPOD_TEMPLATE_ID" | jq .
```

### Important: `prevent_destroy`

All three bucket resources have `prevent_destroy = true`. Terraform will
refuse to delete them even on `terraform destroy`. To actually delete a bucket:
1. Remove from state: `terraform state rm minio_s3_bucket.audio_raw`
2. Delete manually: `aws s3 rb s3://kural-audio-raw --endpoint-url $AWS_ENDPOINT_URL`

## Ansible — Server Configuration

`infra/ansible/site.yml` runs on a fresh server and sets it up completely.
It is idempotent — safe to run multiple times.

### Phases

| # | Phase | Tags |
|---|---|---|
| 1 | System updates + base packages (curl, wget, git, jq, htop, fail2ban, UFW) | — |
| 2 | UFW firewall — deny all inbound except 22, 80, 443 | — |
| 3 | SSH hardening — key-only auth, no password login | — |
| 4 | Docker — installed from official Docker repo | — |
| 5 | Volume directories — `/opt/kural`, `/mnt/kural-data/kural/{db,uploads,logs}` | — |
| 6 | Sync app files — rsync `src/`, `docker/`, `requirements/` to `/opt/kural/` | `deploy` |
| 7 | Write `.env` — secrets from Ansible vars → `/opt/kural/.env` (mode 0600) | `deploy` |
| 8 | Build + start — `docker compose build api` then `docker compose up -d` | `deploy` |
| 9 | Verify — health check on `http://localhost:8000/health` | `deploy` |

**Using tags for fast redeploy:** Phases 1–5 only need to run once. For code changes, use `--tags=deploy` (or `deploy.sh sync`) to run only phases 6–9:

```bash
# Inside ops container
ansible-playbook -i /ops/ansible/inventory.yml /ops/ansible/site.yml --tags=deploy

# Or via deploy.sh
./scripts/deploy.sh sync
```

## Networking & Costs

### Server costs (approximate monthly)

| Resource | Type | Cost/mo |
|---|---|---|
| CPU server | cx23 (2 vCPU, 4 GB) | ~€4.50 |
| Persistent volume | 50 GB | ~€2.50 |
| Object storage | per GB stored | ~€0.01/GB |
| **Total fixed** | | **~€7.50/mo** |

### RunPod costs

Pay-per-second of GPU time. No cost when idle.

- RTX 4000 Ada (~$0.24/hr) processing ~20 min/job → ~$0.08/job
- Idle between jobs → $0

### Egress note

The server and object storage are both in `nbg1` (same network zone).
Data transfer between them is **free**. Egress only costs when data
leaves Hetzner's network (e.g., to RunPod for audio download, or to end users).

## Troubleshooting

### Terraform: "bucket already exists"
```bash
terraform import minio_s3_bucket.audio_raw kural-audio-raw
```

### Terraform: "SSH key not unique"
```bash
hcloud ssh-key list        # note the ID
terraform import hcloud_ssh_key.kural <ID>
```

### Terraform: "server type cx22 not found"
Hetzner renamed their server types. Use `hcloud server-type list`.

### S3: "Invalid region"
`AWS_DEFAULT_REGION` must be a simple string like `nbg1`, not a URL.

### S3: "NoCredentials"
The `AWS_*` variables aren't set in the container. Check `infra/.env.ops`.

### RunPod worker goes unhealthy immediately
1. Check the worker logs in RunPod Console → Serverless → your endpoint → Logs
2. Common causes:
   - GHCR registry credential expired → create new PAT, update RunPod credential
   - Wrong image name (uppercase) → ensure lowercase in template
   - `runpod` SDK version too old → pin `runpod>=1.7,<2` in `requirements/worker.txt`
   - `use_auth_token` vs `token` kwarg mismatch in PyAnnote → set `HF_TOKEN` env var, don't pass token kwarg to `Pipeline.from_pretrained()`

### RunPod worker stuck in back-off loop
On the endpoint page → click ⋯ → Edit Endpoint → Save Endpoint.
This clears the back-off and triggers a fresh start.

### docker-compose: "variable is not set"
The `environment:` section in docker-compose uses `${VAR}` which reads from
the **host shell**, not from `env_file`. All mappings should be in `.env.ops`
directly with the final variable names — no shell references.

## Security Notes

- All secrets are in `infra/.env.ops` — **gitignored, never committed, never pasted in notes.md**
- SSH deploy key (`infra/deploy-keys/kural_ed25519`) is committed — **safe because this is a private repo**. If the repo ever becomes public, rotate the key immediately
- Server SSH: key-only authentication, no password login
- Firewall: only ports 22/80/443 open (Hetzner + UFW double layer)
- Fail2ban: active on SSH
- S3 buckets: all private, no public access
- Worker image: private GHCR; RunPod pulls using stored PAT credential
- HuggingFace token: passed as BuildKit secret at build time — not stored in any image layer
- Terraform state (`*.tfstate`): **gitignored** — store remotely (S3 backend) for team use
